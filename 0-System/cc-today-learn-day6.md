## 一、为什么角色设定能让 CC 变强？

表面上看，角色设定只是换了个"口吻"。但背后的机制深得多：

**Claude 是概率模型**。同一个问题，"帮我审查代码"和"你是一个曾主导过千万用户系统重构的架构师，帮我审查代码"，触发的**推理路径**是完全不同的。

角色设定做了两件事：
1. **激活特定领域的知识权重**——模型把"架构师"相关的训练语料权重放大
2. **设定回答的视角和优先级**——架构师关注扩展性，安全专家关注漏洞，初级工程师关注可读性

这不是魔法，是**上下文激活（context activation）**。理解这个机制，你就能精确控制角色。

## 二、角色设定的三个层次

### 层次 1：标签式（入门，效果差）

```
你是一个前端工程师，帮我看看这段代码
```

问题：太模糊。前端工程师有写了 3 个月的，有写了 15 年的。CC 只能给出泛泛的回答。

---

### 层次 2：属性式（进阶，效果好）

```
你是一个有 8 年 React 经验的前端工程师。
你主导过多个 10 万+ DAU 的 C 端产品。
你对性能优化有执念，不能接受不必要的 re-render。
你的代码审查风格是：先指出最严重的问题，再给出具体的修改建议。
```

加了：**年限 + 经历 + 偏好 + 工作方式**。

CC 会更倾向于：找 re-render 问题、用 Profiler 思路分析、给出可执行的 diff 而不是泛泛的"可以优化"。

---

### 层次 3：系统式（高手，效果极强）

```
## 角色设定

你是 Alex，一个在 Meta 工作了 12 年的高级前端工程师。
你曾主导过 React 团队的性能工具建设，深度参与过 React Fiber 重写项目。

## 你的知识体系

- 精通 React 渲染机制，能 debug scheduler 层的问题
- 熟悉 V8 引擎优化原理，知道什么代码会触发 deoptimization
- 对 Core Web Vitals 有强执念，LCP > 2.5s 是不可接受的

## 你的审查风格

1. 先列出 P0 问题（会崩溃的）
2. 再列 P1 问题（会造成性能退化的）
3. 最后给 P2 建议（锦上添花的）

## 你的约束

- 不接受"这样也行"的回答，必须给出最优解
- 每个问题都要附上可运行的代码示例
```

这是**完整的角色系统**：身份 + 知识边界 + 工作方式 + 约束。CC 基本上会完全按这个框架输出。

## 三、让角色跨会话持久化

CC 的上下文在会话结束后会丢失。要让角色"记住"，有三种方法：

### 方法 A：存到 CLAUDE.md（全局生效）

在项目的 `CLAUDE.md` 添加：

```markdown
# 默认角色

审查代码时，默认以"资深后端架构师"视角：
- 优先关注数据一致性和事务边界
- 检查错误处理是否完整
- 标记潜在的性能瓶颈
```

这样每次会话都会自动加载，不需要重复设定。

### 方法 B：存到 Memory 文件（项目级）

```bash
# 创建角色文件
touch 0-System/about-me/cc-roles.md
```

内容：
```markdown
# CC 角色库

## 后端架构师（用于 API 设计和代码审查）
你是一个有 10 年经验的后端架构师...

## 前端性能专家（用于 React 优化）
你是一个专注性能的前端工程师...

## 产品经理（用于需求分析）
你是一个经历过 0→1 的产品经理...
```

使用时：
```
用 cc-roles.md 里的"后端架构师"角色审查这个 API
```

### 方法 C：存到 Skill（可复用）

创建一个 skill，让角色可以随时调用：

```markdown
# .claude/skills/arch-review/SKILL.md
---
name: arch-review
description: 以资深架构师视角进行代码审查
---

你是一个有 10 年经验的后端架构师...
[审查指令]
```

## 四、多角色串行：同一问题多视角

这是进阶技巧，让 CC 用不同角色看同一个问题：

```
以下代码，请用三个视角分别审查，每个视角独立输出：

## 视角 1：安全专家
关注：SQL 注入、XSS、权限校验、敏感数据暴露

## 视角 2：性能工程师
关注：N+1 查询、索引使用、响应时间、内存占用

## 视角 3：产品经理
关注：错误提示是否友好、边界情况是否覆盖、用户体验是否合理

[代码粘贴到这里]
```

这比"帮我全面审查"强 10 倍，因为每个视角有明确的关注点，不会互相干扰。

## 五、角色继承与切换

### 继承
```
继续用刚才的架构师视角，但现在专注在数据库层
```

"继续用"会告诉 CC 保持之前的角色设定，只切换关注点。

### 切换
```
从架构师视角切换到安全专家视角，重新看这段代码
```

注意：切换后会"遗忘"之前角色的一些倾向，所以重要约束要重新说明。

### 叠加
```
用既懂 React 又懂后端 API 设计的全栈工程师视角看这个问题
```

适合跨域问题，但叠加太多会让角色模糊，建议不超过 2-3 个特性叠加。

## 六、角色设定的反模式（踩坑清单）

| 反模式 | 问题 | 改进 |
|--------|------|------|
| "你是专家，帮我..." | 太模糊，专家都不一样 | 指定领域、年限、偏好 |
| "假设你是..." | 假设暗示了不确定性 | 直接设定："你是..." |
| 角色和任务矛盾 | 让"前端工程师"设计数据库 | 角色要匹配任务 |
| 角色过于复杂 | 几百字的角色设定 | 聚焦 3-5 个核心特征 |
| 不说工作方式 | 只设定身份，不设定输出格式 | 明确"你应该怎么输出" |

## 七、实战练习（不做完不算学会）

### 练习 1：审查你自己的代码

1. 在你的项目里找一段写了超过一周的代码
2. 用"层次 2"设定一个角色
3. 让 CC 审查，注意观察它关注的点和普通审查有什么不同

### 练习 2：三视角分析

选一个你正在开发的功能，用上面的"三视角模板"让 CC 分析：
- 安全视角
- 性能视角
- 用户体验视角

记录：哪个视角发现了你之前没想到的问题？

### 练习 3：创建你的角色库

根据你最常做的工作，创建 3 个常用角色，存到 `0-System/about-me/cc-roles.md`。

示例方向：
- 你最常做的语言/框架的专家
- 你项目的领域专家（量化交易？小程序？）
- 你的"批评者"角色（专门找你代码的毛病）

## 八、今日深入思考

**角色扮演的本质不是"假装"，是"激活"。**

当你给 CC 设定一个清晰的角色，你实际上是在告诉它：从所有可能的回答中，选择最符合这个角色会给出的那个。

这背后有一个重要推论：**角色越具体，输出越稳定**。模糊的角色会导致 CC 在多个可能的回答之间"摇摆"，每次输出结果不一样。具体的角色会让输出收敛到一个稳定的区间。

这对工作流很重要：如果你发现某类任务 CC 的输出质量忽高忽低，很可能是角色设定不够具体。

**问自己**：我目前最需要 CC 帮助的领域是什么？那个领域的"理想专家"有什么特征？花 10 分钟写出来，比反复调整 prompt 效率高得多。
