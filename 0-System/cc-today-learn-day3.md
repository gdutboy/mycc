# Day 3 - 代码审查：让 CC 帮你找问题

> 目标：把 CC 变成 24 小时待命的专属代码审查员

---

## 为什么用 CC 审查？

| 人工审查 | CC 审查 |
|---------|---------|
| 需要等人 | 随时可用，几秒出结果 |
| 心情影响 | 标准一致 |
| 会疲劳 | 不知疲倦 |
| 只能看当前文件 | 会自动读相关文件 |

---

## 审查类型详解

### 类型 1：Bug 查找

**适用场景**：修 bug 前、代码看起来有问题

**提示词**：
```
审查这个文件，找出所有可能导致错误的代码问题：
1. 逻辑错误
2. 空指针/空值问题
3. 边界条件遗漏
4. 异步处理问题
```

**案例**：
```javascript
// CC 会指出问题
function getUser(id) {
  const user = db.find(id);
  return user.name; // 如果 user 是 undefined，这里会报错
}
```

### 类型 2：安全审查

**适用场景**：上线前、涉及用户数据

**提示词**：
```
用安全专家的标准审查这个登录模块：
1. 是否有 SQL 注入风险
2. 密码是否安全处理
3. 是否有 XSS 漏洞
4. 认证和授权是否正确分离
```

**案例**：
```javascript
// CC 会警告
const query = `SELECT * FROM users WHERE id = ${userId}`;
// SQL 注入风险！
```

### 类型 3：性能审查

**适用场景**：优化前、接口响应慢

**提示词**：
```
审查这段代码的性能问题：
1. 是否有 N+1 查询
2. 是否有重复计算
3. 是否有不必要的重渲染
4. 是否有内存泄漏风险
```

### 类型 4：代码质量

**适用场景**：PR review、日常审查

**提示词**：
```
审查这个文件的质量：
1. 是否有重复代码
2. 命名是否清晰
3. 函数是否过长
4. 是否有缺失的注释
5. 错误处理是否完整
```

---

## 高级审查技巧

### 1. 指定审查标准

```
用以下标准审查代码：
- 可读性：命名是否自解释
- 可维护性：改动是否需要改多处
- 性能：时间/空间复杂度
- 安全：输入验证、权限检查
```

### 2. 上下文感知

```
审查 auth.ts
（CC 会自动读相关的 middleware、utils、config）
```

### 3. 搜引用

```
这个变量在哪里被使用了？有没有遗漏的地方？
```

CC 会用 Grep 搜一遍，列出所有使用位置。

## 今日思考

> 好的审查不是挑错，而是帮助团队写出更好的代码。用 CC 做日常审查，人工做关键决策——这是最高效的协作方式。

**本阶段验收标准**：能够根据不同场景选择合适的审查类型，熟练使用审查提示词模板。
