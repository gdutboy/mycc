### PACEflow stop hook 误报修复

- **现象**：stop hook 报 "findings 活跃区有 1 个已解决详情段落未归档"，实际是模板示例被误判
- **根因**：两层叠加
  1. `getArtifactDir` 不检查 `.pace/` 子目录，回落到 `projects/mycc/` 模板文件
  2. 模板 HTML 注释里的 `### [日期] 标题` 示例被正则匹配，误判为 stale
- **修复**：
  - `pace-utils.cjs`：`getArtifactDir` + `isPaceProject` 新增 `.pace/` 子目录检查
  - `stop.cjs`：T-383 检查前剥离 `<!-- -->` HTML 注释
  - 删除 vault 和本地 `projects/mycc/` 模板文件
- **教训**：正则匹配文件内容时，必须先剥离 HTML 注释，否则模板示例会触发误报

### nanobot + Seed 2.0 国产方案对比

- **来源**：Datawhale 公众号文章《我搭了一套国产的小龙虾方案》
- **方案**：nanobot（4000行轻量框架）+ Seed 2.0（火山引擎）+ Claude Code，tmux 多会话并行
- **成本**：40 元/月（火山引擎 Lite 套餐）
- **mycc 优势**：远程控制（飞书/网页）、多引擎并行、PACE 质量保障、Windows 原生
- **可借鉴**：
  - 火山引擎 API 兼容 Anthropic 协议，team-mode 可加 Seed 引擎做低成本选项
  - tmux 持久化方案，长时间任务比 spawn 更稳
  - "PRD 是契约"思路与 PACE 一致，可强化

## 反思

agent-bridge 用最少代码实现了核心功能，体现了"小而美"的设计哲学。mycc 在功能上更强大，但分发门槛高。两者互补，值得把分发能力补上。
