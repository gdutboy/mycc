# Day 1 - 基础对话：自然语言任务下达

> 目标：掌握"描述 what 而不是 how"的提问艺术

## 核心原理

CC 的底层逻辑是**理解你的意图**，而不是执行固定命令。你不需要知道怎么实现，只需要说清楚想要什么结果。

**这才是正确的协作方式**：你描述目标，CC 思考路径。

---

## 为什么不要告诉 CC 怎么写？

当你说出"用 useState、用 useEffect"时，你已经替 CC 做了它最擅长的事——思考实现方案。CC 的价值在于：
- 根据你的描述自己选择技术栈
- 发现你没想到的边界情况
- 给出你不知道的更好方案

---

## 四步提问法（附详细案例）

### 场景 1：做一个博客系统

**❌ 初级问法（告诉了怎么写）**：
```
帮我用 React 写一个博客组件，用 useState 管理状态，用 useEffect 读取数据
```

**✅ 高手问法（只说要什么）**：
```
我需要一个博客文章列表页面，要展示：
1. 文章标题和摘要
2. 作者头像和名字
3. 发布日期
4. 用户能点击查看全文
```

**CC 的思考**（这就是你该交给它做的事）：
- 选择合适的 UI 库
- 设计数据获取方案
- 处理加载/错误状态
- 考虑分页或无限滚动

---

### 场景 2：修复一个 Bug

**❌ 初级问法**：
```
帮我看看 auth.ts 第 30 行的 if 语句有什么问题
```

**✅ 高手问法**：
```
用户反馈登录后一段时间就自动退出了，可能和 token 过期有关。请帮我分析可能的原因。
```

**区别**：你告诉了"问题在哪"vs描述"现象是什么"。后者让 CC 全面分析，可能发现你不知道的根因。

---

## 上下文继承

CC 会记住对话上下文，利用这一点：

```
# 第一次对话
我在做一个电商后台，需要处理订单

# 后续对话（不需要重复上下文）
帮我加一个订单列表页面，能按状态筛选

# CC 知道你在做电商，会考虑：
# - 订单状态有哪些
# - 筛选怎么做
# - 表格要显示什么字段
```

## 今日练习（必做！）

### 练习 1：体验"描述结果"
对 CC 说：
```
帮我写一个计算器，只要能加减乘除就行
```
观察 CC 是如何自己决定技术方案的。

### 练习 2：体验"全面描述"
对 CC 说：
```
帮我写一个用户注册功能，要验证邮箱格式，密码至少8位，返回友好的错误提示
```
观察 CC 是否考虑了你没说的边界情况。

### 练习 3：体验上下文
对 CC 说：
```
我在做一个笔记应用（设定上下文）
（不说应用类型）帮我加一个标签管理功能
```
观察 CC 是否理解了你的上下文。

---

## 今日思考

> 描述"what"而不是"how"，本质是把思考路径的时间留给自己。高手提问越简单，CC 发挥空间越大。

**本阶段验收标准**：能够用自然语言描述需求，让 CC 自己推导实现方案，而不是指挥 CC 怎么写代码。
