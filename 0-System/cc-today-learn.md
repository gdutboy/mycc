## 一、为什么“分步”不是形式主义？

很多人以为分步指令只是把一句话拆成三句话，本质没变化。这个理解不对。

在真实开发里，任务失败通常不是因为“不会写代码”，而是因为：

1. 需求理解错了（方向错）
2. 改动范围失控（改太多）
3. 中途没检查（越改越偏）
4. 验证不完整（以为好了，实际没好）

分步指令的价值，就是把这 4 个高频失败点拆成可控节点。你不是在“让 CC 慢一点”，你是在“给复杂任务加护栏”。

一句话总结：

**分步指令 = 把一次不可控的大跳，变成多个可验证的小步。**

## 二、分步执行模型：从“直接做”升级为“状态机执行”

推荐用 5 阶段模型处理中大型任务：

### 阶段 1：探索（Explore）
目标：确认代码位置、现状和边界，不改代码。

示例指令：

```text
先只做探索：
1) 找出和用户登录相关的文件
2) 说明当前登录流程（入口、校验、状态存储）
3) 列出潜在改动点
不要修改任何文件，先给我结论。
```

### 阶段 2：方案（Plan）
目标：明确改哪些文件、每步做什么、风险是什么。

示例指令：

```text
基于你刚才的探索结果，输出实施计划，包含：
- 文件级改动清单
- 每一步要做什么
- 验证方式
- 回滚点
计划写完先停，等我确认。
```

### 阶段 3：执行（Execute）
目标：严格按计划改动，避免“顺手重构”。

示例指令：

```text
按已确认计划执行，只做第 1 步。
完成后展示变更摘要与关键 diff，暂停等待确认。
```

### 阶段 4：验证（Verify）
目标：用命令、测试、场景验证，不靠主观判断。

示例指令：

```text
现在只做验证：
- 跑单元测试
- 跑 lint
- 给出手动验证步骤和结果
不要进入新改动。
```

### 阶段 5：收尾（Wrap-up）
目标：产出可交接结果（变更说明、影响面、后续建议）。

示例指令：

```text
输出最终交付说明：
1) 改了什么（按文件）
2) 为什么这么改
3) 验证证据
4) 剩余风险与后续建议
```

## 三、三种高强度分步模板（可直接复用）

## 模板 A：Bug 修复模板（稳）

```text
我们按 4 步修 bug，严格顺序执行：

第1步：复现
- 描述复现路径
- 给出最小复现条件
- 不修改代码

第2步：定位
- 锁定根因
- 给出证据（调用链/日志/代码点）
- 不修复

第3步：修复
- 仅修改根因相关代码
- 禁止顺手重构
- 给出 diff 摘要

第4步：验证
- 跑相关测试
- 列边界场景
- 输出“是否彻底修复”的结论

每步结束都停下来等我确认。
```

适用：线上问题、回归 bug、复杂业务缺陷。

---

## 模板 B：新功能开发模板（准）

```text
目标：新增“批量导出订单”功能。
请按以下阶段执行：

1) 需求拆解
- 用户入口在哪里
- 参数和权限规则
- 成功/失败态定义

2) 接口设计
- 请求/响应结构
- 错误码
- 幂等与限流策略

3) 前后端实现计划
- 后端文件改动
- 前端文件改动
- 测试用例改动

4) 分批实现
- 先后端，再前端，再测试
- 每批完成停下来汇报

5) 验证
- 正常流、异常流、权限流
```

适用：涉及前后端联动、接口新增、跨模块需求。

---

## 模板 C：重构模板（控风险）

```text
准备重构模块 X，目标是降低复杂度，不改业务行为。
按以下步骤执行：

1) 建立行为基线
- 列出现有行为清单
- 补关键测试（若缺失）

2) 设计重构切片
- 切成 3~5 个可独立提交的小步
- 每步都可运行、可回滚

3) 逐片重构
- 一次只动一个切片
- 每片都跑测试

4) 对比验证
- 重构前后行为一致性
- 性能是否退化
```

适用：核心模块重构、遗留代码治理。

## 四、实战例子：给 Next.js 项目加“操作审计日志”

需求：后台管理系统要记录管理员关键操作（删用户、改权限、导出数据）。

### 错误做法（一次性指令）

```text
帮我加审计日志功能。
```

结果常见问题：
- 直接开改，没先定义“哪些操作要审计”
- 后端写了一半才发现前端没传 operator 信息
- 最后只能凭感觉说“差不多好了”

### 正确做法（分步指令）

```text
按以下步骤执行，逐步停顿：

第1步（探索）：找出现有管理端关键操作 API，列出可接入审计的点。
第2步（设计）：定义审计字段（actor/action/resource/result/timestamp/ip）。
第3步（实现后端）：新增写审计日志逻辑，只改后端并给 diff 摘要。
第4步（实现前端）：补充必要上下文字段传递。
第5步（验证）：构造 3 个场景（成功删除、权限拒绝、导出失败），确认日志完整性。
每一步完成后先汇报，不要直接下一步。
```

这种方式的关键收益：
- 第 2 步先锁数据模型，避免后面返工
- 第 3/4 步分离，问题定位更快
- 第 5 步有“场景验收”，不是主观“看起来可用”

## 五、分步执行的高级控制技巧

### 技巧 1：强制“只做当前步”

给出硬约束词：
- “只做第 N 步”
- “禁止进入下一步”
- “必须等待确认”

这是防止模型“过度积极”的关键。

### 技巧 2：每步固定输出格式

例如要求每步都输出：

```text
[步骤编号]
目标：
动作：
结果：
证据：
下一步建议：
```

这样你每一步都能快速判断是否可放行。

### 技巧 3：加入“失败分支”

不要只写 happy path，要提前约定异常处理：

```text
如果第 2 步发现需要数据库迁移，先停下来，给迁移方案和风险，不要直接执行。
```

### 技巧 4：把“验证”变成硬门槛

```text
若测试失败，禁止声明完成，先给根因和修复计划。
```

这能显著减少“假完成”。

## 六、常见误区（高频踩坑）

1. **分步太粗**：比如“先分析再实现”，但分析里没有明确产出。
2. **分步太细**：拆成 30 步，沟通成本爆炸。
3. **没有暂停点**：写了分步，但没要求“每步停顿确认”。
4. **没有验收标准**：步骤做完了，但没有判定是否通过的标准。
5. **默认允许顺手优化**：导致改动范围失控。

经验值：
- 中等任务拆 4~6 步最稳。
- 每步都要有“可见产物”（清单、diff、测试结果）。

- 输出重构前后复杂度对比（至少是逻辑分支数或函数长度）

## 八、今日思考

你真正要掌握的，不是“如何让 CC 更聪明”，而是：

**如何设计一个不会失控的执行系统。**

分步指令的本质是“控制论”：
- 先定义状态
- 再设置检查点
- 最后用反馈修正偏差

这套方法不仅对 CC 有效，对你管理人类团队也一样有效。一个工程负责人如果能把任务拆到“每一步都可验证”，交付质量会显著提升。

给你的思考题：

**你当前最容易失控的任务类型是什么？是需求理解、实现过程，还是验收环节？**

找到那个最薄弱环节，先用分步模型补上。
