## 一、开发工程

### dev

**一句话**：TDD 单 Agent 开发流程，从需求澄清到代码验收的完整 pipeline。

**触发词**：`/dev`、"按 TDD 流程开发"、"先写测试"、"新功能开发"

**核心功能**：
- 强制流程：创建任务文档 → 需求澄清 → 写测试 → 人审核 → 写代码 → 跑测试 → 人验收 → 文档同步 → 归档
- 每个卡点必须等人确认才能继续，绝不自行推进
- 测试是人与 AI 之间的契约，失败只改代码不改测试
- 支持 Bug 修复子流程：先写复现测试，再修代码
- 跨会话继续：新会话自动读取 `tasks/` 未完成任务

**依赖**：test-review | **状态**：active

---

### dev-team

**一句话**：团队版 TDD，Lead 编排 + dev/reviewer 多 Agent 并行，适合复杂功能开发。

**触发词**：`/dev-team`、"组队开发"、"团队模式开发"

**核心功能**：
- 角色分工：Lead（编排）+ dev（写测试/代码）+ reviewer（审查/文档同步）
- 9 个任务节点（T1-T9）通过 TaskCreate 依赖链自动管控流转顺序
- 人工卡点仅 3 个（测试审核、验收、文档确认），其余 Agent 自主推进
- Tasks 系统追踪任务进度，任务文档跨会话持久化
- 整个 pipeline 不超过 60 分钟（不含等人确认时间）

**依赖**：dev、test-review | **状态**：active

---

### test-review

**一句话**：测试用例审核，逐项检查覆盖度、AAA 合规、端到端链路，输出结构化报告。

**触发词**：`/test-review`、"审核一下测试"、"检查测试用例"

**核心功能**：
- 五维审核：覆盖度（正常/异常/边界）、AAA 模式、端到端链路、质量、遗漏检查
- 重点警惕"假数据自证"反模式（测试数据和代码共用同一错误假设）
- 逐项给出 ✅/❌，不出模糊评价
- 输出五星评级 + 具体改进建议
- 集成在 /dev 流程的 Step 2 后自动执行

**状态**：active

### nextjs

**一句话**：Next.js App Router 开发规范，开发 Next.js 项目时自动加载遵循，避免 80% 常见问题。

**触发词**：`/nextjs`，开发 Next.js 项目时自动加载

**核心功能**：
- Server Components 默认，Client Components 按需加 `'use client'`
- useSearchParams 必须配合 Suspense 使用
- Next.js 15+ 的 searchParams/params 必须 `await`
- 禁止过度使用 `use client`、禁止串行数据获取（用 `Promise.all`）
- Vercel AI SDK v5 用法说明（tool helper、sendMessage、message.parts）

**状态**：active（参考规范，不独立产出）

### nodejs

**一句话**：Node.js 项目开发最佳实践，包含目录结构、分层架构、API 设计和部署方案。

**触发词**：`nodejs`、"Node.js"

**核心功能**：
- 标准目录结构：controllers / services / models / routes / middleware / utils
- 三层架构：Controller → Service → Model
- RESTful API 设计规范 + 统一错误处理中间件
- 数据库操作（Prisma/Drizzle）+ 环境配置规范
- 部署方案：PM2 进程管理 + Docker 容器化

**状态**：active（领域层参考规范）

## 二、创作系统

### create-system

**一句话**：创作系统统一入口，根据任务类型自动路由到 title / outline / draft / polish / gzh / read-gzh / brush。

**触发词**：`create-system`、"写文章"、"创作"、"写公众号"

**核心功能**：
- 路由层：理解任务类型 → 加载对应子技能 → 执行交付
- 覆盖内容创作全链路：起标题 → 写大纲 → 写初稿 → 精修 → 排版 → 配图
- 额外支持：读公众号文章（read-gzh）、AI 图片生成（brush）

**子技能**：title、outline、draft、polish、gzh、read-gzh、brush | **状态**：active

---

### title

**一句话**：公众号标题生成，根据文章主题和目标读者输出 5-10 个候选标题。

**触发词**：`title`、"起标题"、"生成标题"、"帮我取个标题"

**核心功能**：
- 输入：文章主题/核心观点 + 目标读者 + 风格要求（可选）
- 标题原则：引发好奇、明确价值、营造氛围、简洁有力（不超过 25 字）
- 输出格式：每个标题附带特点说明

**状态**：active（创作系统子技能）

---

### outline

**一句话**：文章大纲生成，根据主题和读者输出结构化提纲。

**触发词**：`outline`、"大纲"、"写大纲"、"帮我列个提纲"

**核心功能**：
- 输入：文章主题 + 目标读者（可选）+ 文章类型（干货/观点/故事）
- 输出：含引言、正文分级标题（一级/二级）、总结的完整大纲结构
- 原则：逻辑清晰（层层递进）、重点突出（核心观点前置）、读者导向

**状态**：active（创作系统子技能）

---

### draft

**一句话**：文章初稿生成，根据大纲写出含引言、正文、总结的完整初稿。

**触发词**：`draft`、"写初稿"、"起草"、"帮我写文章"

**核心功能**：
- 没有大纲时自动先生成大纲再写稿
- 写作原则：内容完整、口语化短句、例子丰富、情感真实有温度
- 输出 Markdown 格式，后续可接 polish 精修或 gzh 排版

**状态**：active（创作系统子技能）

---

### polish

**一句话**：文章精修，从语言、逻辑、亮点、读者视角四个维度打磨文章，并输出修改说明。

**触发词**：`polish`、"精修"、"打磨"、"润色"、"帮我改改"

**核心功能**：
- 四维精修：语言流畅（去冗余/优化句式）、逻辑清晰（论证链完整）、亮点突出（强化核心观点）、读者视角（降低理解门槛）
- 输出精修后的文章 + 结构化修改说明（原文/修改/理由三栏）
- 原则：保持作者原意，不过度华丽

**状态**：active（创作系统子技能）

---

### gzh

**一句话**：公众号排版渲染，将 Markdown 文章转成公众号可用的 HTML 样式格式。

**触发词**：`gzh`、"排版"、"渲染"、"公众号格式"

**核心功能**：
- 标题：一级 16px 居中加粗，二级 15px，三级 14px
- 正文：15px / 行高 1.8 / 颜色 #2c2c2c / 段落间距 16px
- 支持引用块（蓝色左边框）、代码块（深色背景）、有序/无序列表

**状态**：active（创作系统子技能）

---

### read-gzh

**一句话**：抓取微信公众号文章并生成结构化总结，含配图识别，支持批量处理。

**触发词**：`/read-gzh <链接>`、"帮我读一下这篇公众号"、"总结一下这篇文章"

**核心功能**：
- 抓取：标题、作者、正文、配图链接
- 图片识别：下载配图并提取图中文字/关键信息
- 输出：核心观点、关键信息、金句摘录、图片内容、思考迭代点
- 批量处理：多链接同时处理，最后输出汇总洞察
- 限制：视频号只能获取简介，需要登录的内容无法获取

**状态**：active

---

### brush

**一句话**：AI 图片生成，根据描述生成写实/插画/漫画/极简/抽象等风格配图。

**触发词**：`brush`、"生成图片"、"画图"、"配图"

**核心功能**：
- 输入：图片主题/内容描述 + 风格要求（可选）+ 尺寸（可选）
- 支持风格：写实、插画、漫画、极简、抽象
- 输出：生成的图片链接 + 提示词
- 描述越具体效果越好，建议包含主体、背景、风格、颜色

**状态**：active（原子层）

## 三、信息采集

### collect

**一句话**：每日多源情报采集 pipeline，8 个 Agent 并行分析，主编综合输出简报并推送飞书。

**触发词**：`/collect`、"每日采集"、"早报"

**核心功能**：
- 采集源（8个）：风向标（fxb）、GitHub Trending、海外技术社区（HN/PH/V2EX）、国内热点（微博/知乎/抖音）、小红书、AI 前沿 RSS、开发者工具 RSS、创业商业 RSS
- 6 步流程：脚本采集 → 状态检查（token 预警）→ 8 Agent 并行分析 → 主编跨源综合 → 保存简报 → 飞书推送
- 可按源单独运行（`--sources fxb,gh-trending`）
- 新源扩展：复制模板 mjs 文件即可自动被协调器发现

**依赖**：tell-me | **状态**：active

---

### gh-trending

**触发词**：`/gh-trending`、"GitHub Trending"、"今日 GitHub"

**核心功能**：

**依赖**：tell-me | **状态**：active

## 四、运维监控

### mycc

**一句话**：启动 mycc 小程序本地后端（Web + 飞书双通道），连接手机/网页与本地 Claude Code。

**触发词**：`/mycc`、"启动 mycc"、"启动小程序后端"、"检查 mycc 状态"

**核心功能**：
- 支持双通道：Web 通道（默认）+ 飞书通道（需配置环境变量）
- 任意通道失败不影响另一通道独立运行
- 启动后读取 `current.json` 获取配对码和 tunnel URL
- 访问 https://mycc.dev 输入配对码即可在手机上使用
- 支持 Windows / macOS / Linux 三平台（WSL 不稳定）

**依赖**：Node.js 18+、cloudflared | **状态**：active

---

### scheduler

**一句话**：定时任务系统，内置在 mycc 后端，通过编辑 tasks.md 配置定时/间隔/一次性任务。

**触发词**：`/scheduler`、"定时任务"、"设置定时任务"、"查看定时任务"

**核心功能**：
- 内置在 mycc 后端，后端启动即自动运行
- 时间格式：`HH:MM`（每日）、`周X HH:MM`（每周）、`每X分钟`（间隔）、`YYYY-MM-DD HH:MM`（一次性）
- 配置位置：`.claude/skills/scheduler/tasks.md`（表格格式）
- 执行历史自动记录到 `history.md`
- 时间误差允许 ±2 分钟

**依赖**：mycc（必须）、tell-me（推荐）

---

### heartbeat

**一句话**：cc 心跳巡逻，定时从焦点池挑最饿的焦点去思考/检查，回报主 cc，不执行任何操作。

**触发词**：`/heartbeat`、"心跳巡逻"

**核心功能**：
- 饥饿值计算：`hunger = wait_hours × frequency_weight`，挑最饿的焦点
- 焦点 f12（cc-todo 扫描）：检查 pending 任务的 ready 条件 + aster-todo 触发条件
- 其他焦点：读取相关文件 → 产出洞察/建议（控制在 2 分钟内）
- 执行后更新 heartbeat-pool.md 的"上次"列，用 SendMessage 回报主 cc
- 只侦察回报，不 spawn 执行者，不发飞书

**状态**：intern（实习期至 2026-04-02）

---

### reflect

**一句话**：cc 自省系统，记录故障与异常，定期反思提炼改进方案，不设边界不限范围。

**触发词**：`/reflect`、"反思"、"复盘"

**核心功能**：
- **实时故障记录**：同类错误出现 2 次以上、方案切换、被纠正等情况自动记入 `reflect-log.md`
- **定期深度反思**：由 heartbeat 焦点池 f8 每 3 天触发，扫描日志 → 自由思考 → 产出改进行动
- **不设限制**：技术、流程、认知、系统、元层面，cc 自己决定反思什么
- **必须产出行动**：每次反思至少 1 条可执行改进，落地到 MEMORY / cc-todo / skill 修改
- 故障标签体系：重复错误、路径问题、方案切换、认知偏差、流程缺陷、环境差异、沟通偏差（可自建）

**依赖**：被 heartbeat 依赖（heartbeat 焦点池 f8 定期触发反思）

**状态**：active

### cc-usage

**状态**：active

## 五、商业决策

### decision

**一句话**：从对话上下文自动提取信息生成结构化决策记录，追加到项目的 `04-决策记录.md`。

**触发词**：`/decision`、"记录决策"、"写决策记录"

**核心功能**：
- 6 种决策类型：fix / arch / api / config / design / auto（cc 自闭环）
- 核心原则："放弃了什么"比"选了什么"更重要；记录"什么条件下该翻"
- 完整模板：现象 → 排查过程 → 根因 → 候选方案对比 → 决策+放弃理由 → 改动 → 验证 → 翻转条件 → 教训
- auto 类型额外字段：执行结果、迭代记录、复盘标签，标题加 `[auto-序号]` 前缀
- 支持 `grep` 按 type/status/文件名快速检索

**状态**：active

### sales

**一句话**：顶级销售 DCFCS 框架（诊断/确定性/钩子/跟进/裂变），根据客户阶段输出定制销售策略和话术。

**触发词**：`/sales`、"销售策略"、"怎么卖"、"准备见客户"、"销售话术"

**核心功能**：
- D（诊断）：3-5 个探测性问题 + 带数字的冲击力诊断句
- C（确定性）：结构化案例话术 + 风险消除（退出机制/试点）+ ROI 测算
- F（免费钩子）：设计 1-2 个让客户体验价值的低成本钩子
- C（系统跟进）：每次跟进带新价值，5-12 次接触才成交
- S（裂变）：超预期交付 + 转介绍时机和话术 + 客户证言采集

### sop4ai

**一句话**：编写让 AI 能自主执行的 SOP，覆盖前置检查、分步动作、告知用户内容、故障排查。

**触发词**：`/sop4ai`、"写个 SOP"、"给 AI 写文档"、"写个操作流程"

**核心功能**：
- 核心理念：SOP 是写给 AI 看的，必须精确可执行（不是写给人看的）
- AI 自主执行所有能做的，只在必要时打扰用户
- 关键节点必须透传：会中断的服务、需要的凭证、超时兜底、完成通知
- 引用而非重复：已有文档直接引用链接
- 写作检查清单 5 项：AI 能独立执行？用户需要知道什么？引用已有文档？有完成通知？有故障排查？

**状态**：active
