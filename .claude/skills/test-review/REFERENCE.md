# test-review 详细参考

## 审核清单详解

### 覆盖度检查

| 检查项 | 说明 |
|--------|------|
| 正常路径（Happy Path） | 最常见的使用场景都覆盖了吗？ |
| 异常路径（Error Path） | 错误输入、网络失败、超时、权限不足等 |
| 边界值（Boundary） | 空值、null、undefined、0、负数、极大值、空字符串、空数组 |
| 并发/竞态（如适用） | 多线程、多用户同时操作的场景 |

### 常见遗漏清单

逐项过一遍：
- [ ] 空字符串 `""`
- [ ] null / undefined
- [ ] 空数组 `[]`
- [ ] 空对象 `{}`
- [ ] 数字 0
- [ ] 负数
- [ ] 超长字符串（超过限制）
- [ ] 特殊字符（emoji、换行符、SQL 注入字符）
- [ ] 并发请求
- [ ] 网络超时
- [ ] 权限不足

## 端到端链路检查详解

> **教训来源**：P08 mycc 活跃会话改名 bug（2026-02-03）
> 测试用假数据自证，代码和测试用了同一个错误格式，互相印证了错误。

**反模式（必须避免）**：
```typescript
// ❌ 错误：手动构造假数据，和代码用了同一个错误假设
it("能读取 customTitle", () => {
  createTestData({ type: "customTitle", customTitle: "测试" });
  // 代码检查：type === "customTitle"
  // 测试通过！但实际写入的是 "custom-title"
});
```

**正确做法**：
```typescript
// ✅ 正确：用真实函数写入，用另一个函数读取
it("端到端：写入后能正确读取", () => {
  writeFunction("测试标题");
  const result = readFunction();
  expect(result.title).toBe("测试标题");
});
```

**检查清单**：
- [ ] 涉及"写入 → 读取"的功能，是否有端到端测试？
- [ ] 测试中是否有 `as any` 类型断言？如果有，检查类型定义是否完整
- [ ] 测试数据的格式是否与真实写入函数的格式一致？

## 输出示例

```markdown
## 测试用例审核报告

### 覆盖度
- ✅ 正常路径：已覆盖（3 个 case）
- ❌ 异常路径：缺少网络超时场景
- ⚠️ 边界值：缺少空字符串测试
- ➖ 并发：不适用

### AAA 合规
- ✅ 所有 case 结构清晰
- ❌ `test_create_user` 的 Assert 不够具体，只检查了 truthy

### 端到端链路
- ✅ 有端到端测试覆盖写入→读取链路
- ❌ 测试数据是手动构造的，未使用真实写入函数

### 质量
- ✅ 命名清晰
- ❌ `test_case_1` 和 `test_case_2` 测的是同一件事

### 遗漏
- ❌ 缺少空输入测试

### 改进建议
1. 补充网络超时测试：`should_timeout_when_network_is_slow`
2. 补充空字符串测试：`should_return_error_when_name_is_empty`
3. 修改 `test_create_user` 断言，改为检查具体字段值

### 评级
⭐⭐⭐☆☆ (3/5) - 基本覆盖，但有明显遗漏
```

## 评级标准

| 评级 | 标准 |
|------|------|
| ⭐⭐⭐⭐⭐ (5/5) | 覆盖完整、质量高、无遗漏 |
| ⭐⭐⭐⭐☆ (4/5) | 覆盖良好、有小问题 |
| ⭐⭐⭐☆☆ (3/5) | 基本覆盖、有明显遗漏 |
| ⭐⭐☆☆☆ (2/5) | 覆盖不足、质量问题多 |
| ⭐☆☆☆☆ (1/5) | 严重不足、需要重写 |

## 使用示例

**场景 1：cc 写完测试后自检**
```
cc: 测试写完了，我先自检一下...
cc: [执行 /test-review]
cc: 发现 2 个问题，已修复。审核报告如下：[报告内容]
    请确认测试用例是否满足需求。
```

**场景 2：人手动触发审核**
```
user: /test-review tests/user.test.ts
cc: [对指定文件执行审核]
cc: 审核完成，报告如下：[报告内容]
```
