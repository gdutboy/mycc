# decision skill — 详细参考

## 核心原则

1. **"放弃了什么"比"选了什么"更重要** — 选了什么看代码就知道，放弃了什么只有决策记录里有
2. **记"什么条件下该翻"** — 约束条件变了，决策可能就该翻了
3. **从对话上下文自动提取** — agent 写草稿，人扫一眼确认，不增加额外负担
4. **兼容现有格式** — 追加到项目的 `04-决策记录.md` 末尾

---

## 标准决策记录模板

```markdown
## [标题]（[日期]）

<!-- type: fix | arch | api | config | design -->
<!-- status: 生效中 | 已替代 by [标题] | 待实施 -->
<!-- scope: 前端 | 后端 | Worker | 服务器 | 全栈 -->
<!-- files: file1.ts, file2.ts -->
<!-- related: [关联决策标题1], [关联决策标题2] -->

### 现象

[用户/系统遇到了什么问题？怎么发现的？]

### 排查过程

[怎么定位到根因的？查了什么日志、读了什么代码？]

### 根因

[一句话说清楚为什么会出这个问题]

### 候选方案

| 方案 | 做法 | 优点 | 缺点/风险 |
|------|------|------|----------|
| A. [名称] | [描述] | [优点] | [缺点] |
| B. [名称] | [描述] | [优点] | [缺点] |

### 决策

**选择**：[方案 X] — [名称]

**理由**：[为什么选这个，核心判断依据]

**放弃的方案**：
- [方案 Y]：[为什么放弃，什么条件下可以重新考虑]

### 改动

[改了哪些文件，每个文件改了什么]

### 验证

[怎么验证的？测试结果？]

### 什么条件下该翻这个决策

- [约束 1]
- [约束 2]

### 教训

[踩了什么坑？以后怎么避免？]
```

---

## 字段必要性

| 字段 | 必须 | 说明 |
|------|------|------|
| 标题 + 日期 | 必须 | |
| HTML 注释 header | 必须 | type/status/scope/files/related |
| 现象 | 必须 | 没有现象就没有决策 |
| 排查过程 | 可选 | fix 类型建议写，design 类型可跳过 |
| 根因 | 必须 | 不写根因的修复是瞎改 |
| 候选方案 | 必须 | **这是决策记录最核心的部分** |
| 决策 + 理由 | 必须 | |
| 放弃的方案 | 必须 | **比"选了什么"更重要** |
| 改动 | 建议 | 方便代码考古 |
| 验证 | 建议 | |
| 什么条件下该翻 | 必须 | **决策的保质期标签** |
| 教训 | 可选 | 有就写，没有不硬凑 |

---

## 按类型裁剪

| 类型 | 重点字段 | 可精简字段 |
|------|---------|----------|
| `fix` | 排查过程、根因、教训 | 候选方案（修法明显只有一种时） |
| `arch` | 候选方案、放弃理由、翻转条件 | 排查过程 |
| `api` | 候选方案、改动、向后兼容 | 排查过程 |
| `config` | 现象、改动、教训 | 候选方案（选项少时） |
| `design` | 候选方案、用户视角、放弃理由 | 排查过程、根因 |
| `auto` | 候选方案、决策理由、执行结果、迭代记录、复盘标签 | 排查过程、根因、改动 |

## HTML 注释 header 说明

```markdown
<!-- type: arch -->
<!-- status: 生效中 -->
<!-- scope: 后端 -->
<!-- files: adapters/official.ts, http-server.ts -->
<!-- related: v2 Session API 改造 -->
```

**status 流转**：`待实施` → `生效中` → `已替代 by [新决策标题]`

被替代时加：`<!-- superseded: true -->`

**agent 搜索方式**：
```bash
grep "<!-- type: fix -->" 04-决策记录.md
grep "<!-- status: 生效中 -->" 04-决策记录.md
grep "official.ts" 04-决策记录.md
```

## auto 类型模板

```markdown
## [auto-{序号}] {标题}（{日期}）

<!-- type: auto -->
<!-- status: 生效中 -->
<!-- scope: {范围} -->
<!-- risk: 🟢/🟡 -->

### 现象

[为什么要做这个决策/行动]

### 候选方案

| 方案 | 做法 | 优点 | 缺点/风险 |
|------|------|------|----------|
| A. [名称] | [描述] | [优点] | [缺点] |
| B. [名称] | [描述] | [优点] | [缺点] |

### 决策

**选择**：[方案 X]
**理由**：[核心判断依据]
**放弃的方案**：[方案 Y]：[为什么放弃]

### 执行结果

[成功/失败/部分成功，具体描述]

### 迭代记录

- Round 1: [做了什么] → [结果] → [判断]
- Round 2: [调整] → [结果]

### 复盘标签

`#可复用` / `#待优化` / `#踩坑`
```

与标准模板的差异：省略排查过程、根因、改动、什么条件下该翻；新增执行结果、迭代记录、复盘标签、risk。

## 示例

用户说 `/decision`，agent 回顾对话生成草稿：

> 我从这次对话中提取了以下决策记录草稿，确认后追加到 P08 的 04-决策记录.md：
>
> ## SDK 子进程崩溃防护（2026-02-23）
> <!-- type: fix -->
> <!-- status: 生效中 -->
> ...
>
> 确认追加？

用户确认后执行追加。

## 资源预算

- 执行时间：≤2 分钟
- 文件修改：1 个文件（04-决策记录.md 追加）
- 对话回溯：当前对话上下文（不跨 session）
