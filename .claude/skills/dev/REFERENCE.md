# dev Skill 参考文档

> 详细说明、模板、例外情况等，供需要时查阅。

---

## 核心原则

- 人类是决策者，cc 是执行者
- 测试用例是精确的需求描述，是人和 AI 之间的契约
- 测试通过 = 代码符合预期
- **测试失败 → 改代码，不改测试**（除非测试本身写错了）
- **每一步都要人确认后才能进入下一步**
- **任务文档是跨会话的上下文载体**

---

## 任务文档模板

文件位置：`tasks/任务名称.md`

```markdown
# 任务名称

> 创建时间：YYYY-MM-DD HH:MM
> 状态：进行中

## 1. 需求澄清

（待填写：确认后的需求描述、边界情况、异常处理）

## 2. 测试用例

（待填写：审核通过的测试代码）

## 3. 实现记录

（可选：复杂任务记录实现思路）

## 4. 文档同步

（待填写：按变更矩阵更新了哪些文档、改了什么）

## 5. 进度

- [ ] 需求确认
- [ ] 测试编写
- [ ] 测试审核
- [ ] 代码实现
- [ ] 测试通过
- [ ] 人验收
- [ ] 文档同步
- [ ] 文档确认

---

*完成后移动到 `tasks/done/` 或删除*
```

## 流程图（详细版）

```
┌─────────────────────────────────────────────────────────────┐
│  Step 0: 创建任务文档                                        │
│  收到需求 → cc 创建 tasks/xxx.md → 记录原始需求              │
├─────────────────────────────────────────────────────────────┤
│  Step 1: 需求澄清                                           │
│  cc 提问 → 来回对话 → 【人确认】→ 写入文档                   │
│                         ↓ 确认后才能继续                     │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 写测试                                             │
│  cc 写测试 → cc 自检(/test-review) → 修复 → 【人审核】       │
│                                               ↓ 确认后继续   │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 写代码                                             │
│  cc 写代码 → 跑测试 → 失败则改代码 → 全绿                    │
│                                      ↓                      │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 人验收                                             │
│  测试通过 → 【人试用】→ 有问题回 Step 3                      │
│                         ↓ 通过                               │
├─────────────────────────────────────────────────────────────┤
│  Step 5: 文档同步                                           │
│  cc 读变更矩阵 → 更新相关文档 → 列出改了什么                 │
│                                   ↓                          │
├─────────────────────────────────────────────────────────────┤
│  Step 6: 人确认文档                                          │
│  【人检查】文档准确 → 归档任务文档                            │
└─────────────────────────────────────────────────────────────┘
```

**关键卡点**（cc 必须等人确认）：
1. ✋ 需求确认后 → 才能写测试
2. ✋ 测试审核后 → 才能写代码
3. ✋ 人验收后 → cc 才做文档同步
4. ✋ 文档确认后 → 才能归档

## 例外情况

| 情况 | 处理 |
|------|------|
| 紧急 hotfix | 先修，后补测试和文档 |
| 纯 UI 调整（颜色、间距） | 可以不写测试，但要有任务文档 |
| 探索性原型 | 先跑通，确定方向后再补测试 |

但事后必须补上测试，不能一直欠着。

---

## 不做的事

- 不跳过人工卡点自行推进
- 不在没有任务文档的情况下开始开发
- 不修改测试来让代码通过（除非测试本身写错）
- 不做架构决策（那是人的事）
