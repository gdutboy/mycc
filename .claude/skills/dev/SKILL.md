# 开发流程规范（TDD Pipeline）

> 测试 = 人和 AI 之间的契约。cc 是执行者，人是决策者。

## 依赖

from .claude/skills/test-review/SKILL.md import /test-review

## 强制流程

```
Step 0       Step 1       Step 2          Step 3      Step 4     Step 5        Step 6
创建任务文档 → 需求澄清 → 写测试+自检 → 写代码 → 人验收 → 文档同步 → 人确认文档 → 归档
              ✋人确认    ✋人审核                  ✋人验收              ✋人确认
```

**每步详细说明：**

- **Step 0**：收到需求立即创建 `tasks/任务名.md`，记录原始需求和进度 checklist
- **Step 1**：主动提问澄清边界、异常情况、边界值；确认后写入任务文档
- **Step 2**：写测试（正常路径 + 异常路径 + 边界），执行 `/test-review` 自检，展示给人审核
- **Step 3**：写代码实现；跑测试，失败则改代码不改测试，直到全绿
- **Step 4**：人手动试用；有问题回 Step 3；通过后进入 Step 5
- **Step 5**：读项目 `00-项目定义.md` 的变更矩阵，更新所有相关文档，列出改了什么
- **Step 6**：人确认文档准确后，任务文档移至 `tasks/done/` 或删除

## 关键禁止

- 不创建任务文档就开始
- 不确认需求就写代码
- 不写测试就写代码
- 测试没人审核就写代码
- 测试没跑通就说"完成了"
- 自作主张跳过人工卡点

## Bug 修复流程

创建任务文档 → 人确认 bug 存在 → 写失败测试 → 人审核 → 改代码让测试通过 → 跑全量测试 → 人验收 → 文档同步 → 归档

## 跨会话继续

新会话开始时检查 `tasks/` 目录，有未完成任务则读取并汇报进度，从上次停下处继续。

## 边界

- 单任务不超过 30 分钟（不含等人确认时间）
- 产出前缀：`[dev]`
