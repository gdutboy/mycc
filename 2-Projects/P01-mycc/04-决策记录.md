# P01 - 决策记录

### [auto-001] 路由治理优先通过 description 收窄边界，而不是继续堆新规则

**日期**：2026-03-13
**类型**：auto

**现象**：
在解释 superpowers 触发词时，出现了 skill 误触发问题，例如 `dashboard` 被用于解释类问题。
同时，多类 skill 的 frontmatter description 边界不清，容易出现“执行类 skill 抢解释类请求”或“泛用 skill 吞掉专用 skill”的情况。

**排查过程**：
1. 回看实际对话，确认问题不是单个 skill 失效，而是整套路由边界定义过宽。
2. 梳理 mycc / superpowers / PACE 三层分工。
3. 对高冲突 skill 逐个检查 description，重点看是否：
   - 只写了“做什么”
   - 没写“不用于什么场景”
   - 没覆盖真实口语触发表达
4. 同时把规则沉淀到正式文档和思考文档，避免后续继续靠口头约定。

**候选方案**：
- 方案 A：继续加更多路由规则和人工判断说明
  放弃理由：规则会越来越多，但触发入口仍然模糊，维护成本高。
- 方案 B：优先收窄 skill description，并补正式路由规范
  采用理由：description 就是第一层触发面，先把入口校准，收益最大，改动最小。
- 方案 C：直接做一整套 should-trigger / should-not-trigger 自动测试
  暂缓理由：价值高，但前提是 description 和路由规范先收口，否则测试集会建立在模糊边界上。

**最终决策**：
优先采取“**文档规范 + description 收窄**”的双轨方案：
1. 新增正式规范文档 `docs/skill-routing-spec.md`
2. 新增思考文档 `3-Thinking/superpowers-mycc-pace-路由设计.md`
3. 先优化高冲突 skill 的 description，明确“该触发 / 不该触发”的边界

**理由**：
这是当前成本最低、收益最高、最容易持续迭代的方案。
先把触发入口校准，后续再补测试集和自动评估，顺序更稳。

**改动文件**：
- `docs/skill-routing-spec.md`
- `3-Thinking/superpowers-mycc-pace-路由设计.md`
- `.claude/skills/dev/SKILL.md`
- `.claude/skills/dev-team/SKILL.md`
- `.claude/skills/skill-creator/SKILL.md`
- `.claude/skills/create-system/SKILL.md`
- `.claude/skills/dashboard/SKILL.md`
- `.claude/skills/scheduler/SKILL.md`
- `.claude/skills/collect/SKILL.md`
- `.claude/skills/decision/SKILL.md`
- `.claude/skills/reflect/SKILL.md`
- `.claude/skills/desktop/SKILL.md`
- `.claude/skills/tell-me/SKILL.md`

**验证结果**：
- 已完成两份文档落地
- 已完成 11 个高冲突 skill 的 description 收口
- 已修复思考文档中的乱码问题
- 当前路由边界相比修改前更清晰，误触发风险下降

**什么条件下应推翻这个决策**：
如果后续发现：
1. description 收窄后出现明显漏触发；
2. 规则文档无法稳定约束实际触发；
3. 多轮真实对话回放后仍然频繁误判；
则应升级到下一阶段：建立 should-trigger / should-not-trigger 测试集，并加入自动评估闭环。

### [auto-002] description 收口完成后，立即补 should-trigger / should-not-trigger 测试集，而不是继续盲目扩规则

**日期**：2026-03-13
**类型**：auto

**现象**：
前一阶段已经完成 `.claude/skills/**/SKILL.md` 真实 frontmatter description 的统一收口，但如果没有基线测试集，后续很难稳定判断：
1. 哪些 skill 仍可能误触发；
2. 哪些边缘表达会漏触发；
3. description 的修改到底有没有带来真实收益。

**排查过程**：
1. 对 `.claude/skills/**/SKILL.md` 做全量扫描，确认真实 frontmatter 已统一为 `Use when ...；不用于 ...`。
2. 识别到 `skill-creator/SKILL.md:70` 的 `description: 做什么 + 什么时候触发` 只是正文示例，不是真实配置。
3. 回看 `docs/skill-routing-spec.md` 与 `3-Thinking/superpowers-mycc-pace-路由设计.md`，确认此前已经把“下一步应补测试集”写成明确后续方向。
4. 基于高冲突 skill 和近似场景，整理第一版 `should-trigger / should-not-trigger` 测试集文档。

**候选方案**：
- 方案 A：继续扩大 description 修改范围
  放弃理由：真实 frontmatter 已基本收口，继续改收益递减，还可能把边界再次搅乱。
- 方案 B：先建立 trigger 测试集，再观察真实对话表现
  采用理由：能把“感觉路由更准了”变成可回放、可对比的基线。
- 方案 C：直接上自动化 eval 闭环
  暂缓理由：方向正确，但先有人工可读测试集，再转结构化 eval 更稳。

**最终决策**：
进入第二阶段：**测试集 + 归档补强**。
1. 新增 `docs/skill-routing-trigger-testset.md`
2. 先用 Markdown 形式沉淀 `should-trigger / should-not-trigger / 边缘样本`
3. 再把项目概览中的下一步切换到“运行测试集 + 观察真实对话”

**理由**：
当前最需要的不是继续加规则，而是给已有规则建立一个可复用的验证基线。
这样后续无论继续改 description，还是升级成自动 eval，都有统一参照。

**改动文件**：
- `docs/skill-routing-trigger-testset.md`
- `2-Projects/P01-mycc/00-项目概览.md`
- `2-Projects/P01-mycc/04-决策记录.md`

**验证结果**：
- 已确认真实 frontmatter description 全部完成收口
- 已区分配置项与示例文本噪音，避免误判漏改
- 已新增第一版 trigger 测试集，覆盖高冲突、高频和边缘表达场景
- 项目后续动作从“继续补 description”切换为“运行测试集并观察效果”

**什么条件下应推翻这个决策**：
如果后续发现：
1. 当前测试集覆盖面太窄，无法反映真实请求；
2. 真实对话中的误触发/漏触发与测试集结论明显不一致；
3. Markdown 测试集已不足以支撑迭代；
则应升级为结构化 eval 数据和自动 benchmark。

### [auto-003] 边缘样本确认后，优先对 `skill-creator` 与 `scheduler` 做最小 description 收口，而不是继续扩大改动范围

**日期**：2026-03-13
**类型**：auto

**现象**：
在第一轮主样本和第二轮边缘样本人工回放后，整体路由边界已基本成立，但仍有两个高风险点：
1. `E04` 中“帮我过一遍这套 skill description 有没有明显漏洞”这类请求，容易让 `skill-creator` 抢走本应属于审阅/分析的场景；
2. `E05` 中“想定期看看 GitHub AI 趋势，但先别创建任务”这类请求，前半句含周期信号，后半句又明确否定执行动作，`scheduler` 存在过早触发风险。

**排查过程**：
1. 回看 `docs/skill-routing-trigger-testset-round1-results.md`，确认主样本整体通过，风险集中在边缘表达。
2. 对 `E02 / E04 / E05` 做第二轮人工判断，进一步区分“最可能落到哪里”而不是只看 description 是否勉强解释得通。
3. 结论是：`dashboard` 当前口语查看请求尚可保持原状；真正值得优先收口的是 `skill-creator` 与 `scheduler` 的否定边界。
4. 因此只对这两个 skill 做 frontmatter description 最小修改，不扩散到正文流程或更多 skill。

**候选方案**：
- 方案 A：继续扩大 description 修改范围，把 `dashboard` 等更多 skill 一并再收窄
  放弃理由：当前证据不足，容易为了防误触发引入漏触发。
- 方案 B：只对 `skill-creator` 和 `scheduler` 做最小文案收口
  采用理由：直接命中当前高风险边界，改动小，验证成本低。
- 方案 C：先不改 description，等待更多真实回放
  暂缓理由：虽然更保守，但 `E04 / E05` 的边界已足够明确，先做小步修正更合适。

**最终决策**：
执行最小收口方案：
1. 在 `.claude/skills/skill-creator/SKILL.md` 中补强排除语义，明确不用于只做审阅、点评、漏洞检查这类分析请求；
2. 在 `.claude/skills/scheduler/SKILL.md` 中补强排除语义，明确不用于只讨论周期需求但尚未要求创建/修改任务的场景；
3. `.claude/skills/dashboard/SKILL.md` 本轮保持不变。

**理由**：
这一步能在不扩大改动面的前提下，优先压住最明显的误触发风险。
先把高风险边界收紧，再进入真实回放观察，比继续泛化改文案更稳。

**改动文件**：
- `.claude/skills/skill-creator/SKILL.md`
- `.claude/skills/scheduler/SKILL.md`
- `docs/skill-routing-trigger-testset-round1-results.md`
- `2-Projects/P01-mycc/04-决策记录.md`

**验证结果**：
- 已完成两处 frontmatter description 最小收口；
- `skill-creator` 新增了对审阅/点评/漏洞检查类请求的排除边界；
- `scheduler` 新增了对“只讨论周期需求、尚未要求创建或修改任务”场景的排除边界；
- `dashboard` 经第二轮边缘样本判断，当前暂不需要修改。

**什么条件下应推翻这个决策**：
如果后续真实回放发现：
1. `E04` 相关表达仍频繁误触发 `skill-creator`；
2. `E05` 相关表达仍频繁误触发 `scheduler`；
3. 或者这次收口导致 `T01`、`T08` 等主样本出现漏触发；
则应重新调整 description，必要时补结构化 eval，而不是继续凭人工感觉微调。

### [auto-004] 边缘样本路由判定与人工判断一致后，先归档为观察结论，不立即升级结构化 eval

**日期**：2026-03-13
**类型**：auto

**现象**：
在前面已经完成 `E02 / E04 / E05` 三条边缘样本的人工判断后，还需要确认：
1. 当前仓库里的真实 skill frontmatter description 是否会把这三条路由到预期位置；
2. 如果结果一致，下一步是立刻升级成结构化 eval，还是先作为观察结论沉淀，等待更多真实用户会话样本。

**排查过程**：
1. 回看 `docs/skill-routing-trigger-testset.md`，确认当前阶段的执行入口是逐条回放测试语句并记录路由结果。
2. 针对 `E02 / E04 / E05` 分别按当前仓库实际 skill 配置做路由判定回放。
3. 将结果补入 `docs/skill-routing-trigger-testset-round1-results.md`，并与“真实用户会话回放”明确分栏，避免混淆。
4. 对比人工判断与当前配置下的路由判定，确认三条样本均一致。

**候选方案**：
- 方案 A：立即把三条边缘样本升级成 JSON / YAML 结构化 eval 基线
  暂缓理由：方向对，但当前还缺真实用户会话样本，过早升级会放大人工判断权重。
- 方案 B：先记录为阶段性观察结论，后续再深化
  采用理由：当前最重要的是把“人工判断与现状配置一致”这个事实沉淀下来，并等待未来真实会话继续验证。

**最终决策**：
选择方案 B：
1. 先把本轮结果记录为阶段性观察结论；
2. 暂不立刻升级结构化 eval；
3. 后续当真实用户会话中再次出现 `E02 / E04 / E05` 类表达时，再继续补真实回放并决定是否升级。

**理由**：
当前已经证明三条高风险边缘样本在现有 description 配置下与人工判断一致，继续马上扩大到结构化 eval 的收益不如先沉淀结论、等待真实样本来得稳。
这样可以避免把“当前模拟路由判定”误当成“真实线上表现”。

**改动文件**：
- `docs/skill-routing-trigger-testset-round1-results.md`
- `2-Projects/P01-mycc/04-决策记录.md`

**验证结果**：
- `E02` 当前配置下判定为触发 `dashboard`，与人工判断一致；
- `E04` 当前配置下判定为未触发业务 skill，与人工判断一致；
- `E05` 当前配置下判定为未触发业务 skill，与人工判断一致；
- 三条样本均已归档为“当前仓库实际 skill 配置下的路由判定回放”，未与真实用户会话回放混写。

**什么条件下应推翻这个决策**：
如果后续出现以下任一情况，应重新考虑立即升级为结构化 eval：
1. 真实用户会话中 `E02 / E04 / E05` 的实际触发结果与当前判断不一致；
2. 相同边缘表达多次出现不稳定路由；
3. 新一轮 description 调整后需要可重复 benchmark 验证；
则应把这批样本转成结构化 eval 数据，而不是继续只靠 Markdown 观察。
