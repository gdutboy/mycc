# AI 协作方法论

> CC 与人共生变强的核心方法论。持续迭代更新。

---

## 一、核心信念

| 信念 | 含义 |
|------|------|
| **文档是给未来的 CC 看的** | 不是给人存档的，是给 AI 理解上下文的数据源 |
| **放弃了什么比选了什么更重要** | 选了什么看代码就知道，放弃了什么只有决策记录里有 |
| **加文件 = 加能力** | 结构化文件就是 CC 的能力单元 |
| **Agent First** | 做任何事都优先让 Agent 去做，人只做审核和方向定义 |
| **共生进化** | CC 越强我也越强，我的经验也随着 CC 进化而增值 |

## 二、项目结构（三层分类）

```
├── 0-System/          # 系统级：状态、上下文、记忆
├── 1-Inbox/           # 临时想法、研究、灵感（未成熟）
├── 2-Projects/        # 正式项目（每个项目有标准结构）
├── 3-Thinking/        # 方法论、认知、判断标准
├── 4-Assets/          # 可复用资产：SOP、模板、prompt
└── 5-Archive/         # 历史归档
```

**使用原则**：
- 想法成熟了 → Inbox → 提升为 Project
- 需要 CC 知道什么 → 放到它能读到的位置
- 按需调用，而非全部注入上下文

## 三、Skill 系统

### 三层架构

```
┌─────────────────────────────────────────────┐
│  路由技能（按需加载）                        │
│  如：dev-team, create-system               │
├─────────────────────────────────────────────┤
│  领域技能（某领域工具集）                    │
│  如：title, gzh, test-review, nextjs        │
├─────────────────────────────────────────────┤
│  原子技能（单一功能）                        │
│  如：tell-me, brush, skill-creator          │
└─────────────────────────────────────────────┘
```

### 路由规则（写进 CLAUDE.md）

```
写代码 → dev / dev-team
写文章 → create-system
信息采集 → collect
复杂任务 → agent team
决策记录 → /decision
```

### Skill 最佳实践

- 不是越多越好：60+ skill 会污染上下文
- 按需加载：通过路由技能而非全部注入
- 某些技能放文档里也能被搜到，不一定做成 Skill

## 四、决策记录

### 为什么重要

CC 能通过 Git 看到文件变了什么，但**不理解为什么变**。决策记录是给未来的 CC 看的上下文恢复文档。

### 必记字段

| 字段 | 必要性 |
|------|--------|
| 背景信息 | 必须 |
| 候选方案 | 必须（核心！） |
| 最终决定 | 必须 |
| 放弃了什么 | 必须（比选了什么更重要） |
| 改动文件 | 建议 |
| 什么条件下该翻 | 必须（决策的保质期） |

### 决策类型

| 类型 | 场景 |
|------|------|
| fix | Bug 修复 |
| arch | 架构变更 |
| api | 接口设计 |
| design | 功能取舍 |
| auto | CC 自闭环决策 |

### 模板

```markdown
## [标题]（日期）

<!-- type: xxx -->
<!-- status: 生效中 -->

### 现象
[遇到什么问题]

### 候选方案
| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. xxx | ... | ... | ... |

### 决策
选择：A 方案
理由：...
放弃的方案：B（因为...）

### 什么条件下该翻
- 约束1变化时
- 约束2变化时
```

## 五、软件开发四个时代

### Era 1：纯人力
```
人 → 写需求 → 写代码 → 测试 → 改 Bug → 部署
```
人既是执行者也是推动者。

### Era 2：人指挥 AI
```
人发指令 → AI 写需求 → 人发指令 → AI 写代码 → ...
```
人变成指挥官，但节点间仍靠人推动。

### Era 3：AI 自驱（当前主流）
```
人给想法 → AI 自动推动 → 写需求 → 写代码 → 测试 → 改 Bug → 部署 → 人验收
```
人只负责起点（想法）和终点（验收），中间全链路 AI 自动推动。

### Era 4：AI 永动（探索中）
```
人写一次功能清单 → 定时唤醒 → AI 自己读清单 → 自己设计 → 自己开发 → 自己验收 → 循环
```
人连启动每一轮都不需要，AI 自己找活干。

## 六、贾维斯模式

### 设计哲学

把自己想象成皇帝，只跟一个贾维斯对话。贾维斯收到任务后，自己去后台起 Agent 执行，不阻塞主对话。

### 核心规则

- **超过 1 分钟的任务** → spawn 子 Agent，主 CC 只编排验收
- **简单问答** → 直接回，不起 Agent
- **需要干活** → 创建 team，后台执行，完成后通知
- **跨机器** → 通过 mycc API 通信

### 效果

- 主 Agent 永远不被阻塞
- 子 Agent 后台跑任务
- 可并行多个任务

## 七、定时心跳系统

每半小时/一小时唤醒一次，从任务池中挑一个执行。

### 示例任务

- 审视最近对话，发现缺什么 Skill → 写入选修池 → 飞书通知
- 扫描 Inbox，挑能推进的想法 → 自己推进
- 采集热点信息
- 扫描过期项催办

### 关键

让 CC 有**主观能动性**：自己扫描、自己判断、自己执行、自己通知。

| 时间 | 项目 | 耗时 |

## 九、壁垒在哪里

官方会不断补齐基础能力（Agent Team、远程、定时任务...）。

**真正的壁垒**：

1. **深入业务的落地经验**（不是技术架构）
2. **个性化 Skill**（业务中打磨出来的）
3. **AI 能力边界认知**（知道什么能做什么不能做）
4. **踩坑经验**（形成直觉和肌肉记忆）

> 官方做的是公约数，服务绝大多数用户。深入具体业务流程的经验，才是你的护城河。

## 十一、核心原则汇总

1. **文档给 CC 看** — 不是给人存档
2. **放弃的比选的重要** — 决策记录核心
3. **结构即能力** — 加文件 = 加功能
4. **Agent First** — 优先让 AI 干活
5. **共生进化** — CC 越强你越强
6. **单一事实来源** — 不散落，避免矛盾
7. **触发条件可执行** — 指令要精确可判断
8. **人决策 AI 执行** — 人管方向，AI 干活
